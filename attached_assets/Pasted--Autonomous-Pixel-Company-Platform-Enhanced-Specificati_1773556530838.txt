## 1. Requirements

### 1.1 Functional

- **Company lifecycle:** Create, configure, pause, resume, archive companies
- **Agent CRUD:** Hire, fire, promote, demote, reassign agents within a hierarchy
- **Goal decomposition:** CEO receives high-level goals; system decomposes into sub-goals cascading down the org tree
- **Parallel heartbeat:** Scheduler fires all due agents concurrently with structured concurrency (not bare Promise.all)
- **Swarm orchestration:** Dynamic specialist fleets with full lifecycle, inter-agent messaging, and synthesis
- **Governance:** Board (user) approves/rejects escalations, hires above threshold, budget overrides
- **Tool system:** Agents can use registered tools (web search, code execution, file I/O, draft comms) scoped by role
- **Memory:** Per-agent short-term (context window) + long-term (vector store or structured KV) memory
- **CLI:** First-class terminal interface for all operations
- **Pixel visualization:** Real-time animated office canvas showing agent states
- **Cost tracking:** Per-call, per-agent, per-company with hard stops and alerts

### 1.2 Non-Functional

- **Latency:** Agent heartbeat initiation < 200ms; SSE event delivery < 500ms
- **Concurrency:** Support 50 simultaneous agent executions per company
- **Cost ceiling:** Hard circuit-breaker at company budget; no silent overruns
- **Portability:** Runs on Cloudflare Workers (prod) or local Node (dev) with same codebase
- **Auditability:** Every LLM call traced with input hash, output hash, cost, latency, model, tokens
- **Resilience:** Agent crash doesn’t cascade; failed agents enter `error` state, not `crashed-the-whole-system` state

### 2.2 Component Responsibilities

#### Heartbeat Scheduler

- **Not** bare `Promise.all`. Uses a **structured concurrency pool** with configurable max parallelism.
- Each agent execution is wrapped in a **circuit breaker** (3 failures → agent enters `circuit_open` state for cooldown).
- **Dead-letter queue:** Failed agent runs are logged with full context for manual retry or governance review.
- **Backpressure:** If the pool is saturated, new heartbeats queue rather than spawn unbounded.

#### Swarm Engine (5-Phase Lifecycle)

```
Phase 1: PROPOSE    — Swarm leader analyzes goal, proposes N specialist roles
Phase 2: APPROVE    — If swarm cost estimate > threshold, requires governance approval
Phase 3: SPAWN      — Ephemeral agents created with scoped prompts, tools, and memory slices
Phase 4: EXECUTE    — Parallel execution with inter-agent message bus (agents can share intermediate results)
Phase 5: SYNTHESIZE — Leader collects all outputs, produces merged result
Phase 6: DISSOLVE   — Ephemeral agents terminated, memory archived, cost finalized
```

#### Governance Gate

- Every action that exceeds an agent’s **capability token** scope triggers a governance request.
- Capability tokens are role-scoped: `ceo.delegate`, `engineer.code.execute`, `marketer.content.draft`.
- Escalation chain: agent → manager → … → CEO → Board (user).
- Governance requests have a **TTL** — if not acted on within N minutes, they auto-reject (fail-safe, not fail-open).

#### Agent Execution Runtime

- **LLM adapter:** Pluggable interface supporting OpenAI, Anthropic, Ollama (local). Model selection per agent.
- **Tool executor:** Sandboxed execution environment. Tools are registered with JSON Schema contracts. Agents can only invoke tools granted by their capability token.
- **Memory manager:** Two-tier: (1) Short-term = last N messages in context window. (2) Long-term = structured KV entries with semantic search (optional vector store).
- **Cost accumulator:** Every LLM call logged with `{model, input_tokens, output_tokens, cost_usd, latency_ms}`. Accumulated per-agent, checked against budget before each call.
- **Trace emitter:** Every agent action emits an OpenTelemetry-compatible span: `agent.heartbeat`, `agent.tool_call`, `agent.llm_call`, `swarm.synthesis`.

### 3.3 Key Design Decisions

|Decision                                          |Rationale                                                                                                    |
|--------------------------------------------------|-------------------------------------------------------------------------------------------------------------|
|ULIDs over UUIDs                                  |Sortable by creation time, no index fragmentation                                                            |
|`version` column on `agent_tasks`                 |Optimistic concurrency: `UPDATE ... WHERE version = $current` prevents double-claiming in parallel heartbeats|
|Separate `heartbeat_runs` / `heartbeat_agent_runs`|Allows per-agent failure without marking entire heartbeat as failed                                          |
|`swarm_messages` table                            |Enables inter-agent communication within swarms (research agent shares finding with implementation agent)    |
|`tool_registry` with `risk_level`                 |High/critical tools auto-trigger governance approval                                                         |
|`capability_token` on agents                      |Serialized scope string like `company.read,goals.write,tools.code_exec` — validated at runtime               |
|JSON `settings` on company                        |Extensible config without schema migrations for every new setting                                            |
|`circuit_breaker` on company                      |Company-wide kill switch when budget is exhausted or anomaly detected                                        |

## 4. Parallel Orchestration Engine (Enhanced)

### 4.1 Structured Concurrency (Not Bare Promise.all)

```typescript
// Conceptual — the actual implementation pattern
class HeartbeatRunner {
  private pool: AgentPool;          // max N concurrent
  private circuitBreakers: Map<string, CircuitBreaker>;

  async runHeartbeat(companyId: string): Promise<HeartbeatResult> {
    const run = await createHeartbeatRun(companyId);
    const dueAgents = await getDueAgents(companyId);

    // Check company-wide circuit breaker first
    if (await isCompanyCircuitOpen(companyId)) {
      return { status: 'skipped', reason: 'company_circuit_open' };
    }

    // Structured concurrency: pool with backpressure
    const results = await this.pool.map(dueAgents, async (agent) => {
      // Per-agent circuit breaker check
      const cb = this.circuitBreakers.get(agent.id);
      if (cb?.isOpen()) {
        return { agentId: agent.id, status: 'skipped', reason: 'agent_circuit_open' };
      }

      try {
        // Budget check BEFORE execution
        if (agent.budget_used_usd >= agent.budget_monthly_usd) {
          return { agentId: agent.id, status: 'skipped', reason: 'budget_exhausted' };
        }

        const result = await executeAgentHeartbeat(agent, run.id);
        cb?.recordSuccess();
        return { agentId: agent.id, status: 'succeeded', result };
      } catch (error) {
        cb?.recordFailure();
        await deadLetterQueue.enqueue({ agent, error, runId: run.id });
        emitEvent('agent.heartbeat.failed', { agentId: agent.id, error });
        return { agentId: agent.id, status: 'failed', error };
      }
    }, { concurrency: 10, timeout: 30_000 }); // max 10 parallel, 30s timeout per agent

    return aggregateResults(run, results);
  }
}
```

### 4.2 Task Claiming (Optimistic Locking)

```sql
-- Atomic task claim: only succeeds if version hasn't changed
UPDATE agent_tasks
SET claimed_by = :agent_id,
    status = 'claimed',
    version = version + 1,
    updated_at = :now
WHERE id = :task_id
  AND status = 'pending'
  AND version = :expected_version;
-- If rows_affected = 0, another agent already claimed it
```

### 4.3 Circuit Breaker State Machine

```
CLOSED ──(3 failures in 60s)──→ OPEN ──(cooldown 300s)──→ HALF_OPEN ──(1 success)──→ CLOSED
                                                           │
                                                           └──(1 failure)──→ OPEN
```

## 5. Swarm Orchestration Engine (Enhanced)

### 5.1 Lifecycle State Machine

```
PROPOSED → PENDING_APPROVAL → SPAWNING → EXECUTING → SYNTHESIZING → COMPLETED → DISSOLVED
    │              │                          │              │
    └──CANCELLED   └──CANCELLED               └──FAILED      └──FAILED
```

### 5.2 Inter-Agent Messaging

Swarm agents communicate via a lightweight message bus (backed by `swarm_messages` table):

```typescript
interface SwarmMessage {
  from: string;       // swarm_agent_id
  to: string | null;  // null = broadcast to all swarm agents
  type: 'finding' | 'question' | 'dependency' | 'partial_result' | 'error';
  content: unknown;
}

// Example: Research agent shares a finding with Implementation agent
await swarmBus.send({
  from: researchAgent.id,
  to: implAgent.id,
  type: 'finding',

});
```

### 5.3 Swarm Cost Estimation

Before spawning, the swarm leader estimates total cost:

```
estimated_cost = N_agents × avg_tokens_per_specialist × cost_per_token × estimated_rounds
```

If `estimated_cost > company.budget_monthly_usd * 0.1` (10% of monthly budget), auto-trigger governance approval.

### 6.3 Hierarchy Enforcement Middleware

```typescript
async function enforceHierarchy(ctx, next) {
  const { actingAgent, targetAgent, action } = ctx;

  // Delegation: can only command direct reports
  if (action === 'delegate') {
    if (targetAgent.manager_id !== actingAgent.id) {
      await createGovernanceRequest({
        type: 'escalation',
        description: `${actingAgent.name} attempted to delegate to non-report ${targetAgent.name}`,
        requesting_agent_id: actingAgent.id,
      });
      throw new HierarchyViolation('Can only delegate to direct reports');
    }
  }

  // Escalation: can only escalate to own manager
  if (action === 'escalate') {
    if (actingAgent.manager_id !== targetAgent.id) {
      throw new HierarchyViolation('Can only escalate to direct manager');
    }
  }

  await next();
}
```

## 9. Security Model

### 9.1 Threat Surface

|Threat                                |Mitigation                                                                                                                                 |
|--------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------|
|**Prompt injection via agent output** |Output sandboxing: agent outputs are treated as untrusted data; never interpolated into system prompts of other agents without sanitization|
|**Privilege escalation**              |Capability tokens validated at every action; hierarchy middleware prevents cross-branch commands                                           |
|**Budget overrun**                    |3-tier budget: per-call check → per-agent cap → company circuit breaker                                                                    |
|**Runaway swarm**                     |Swarm timeout + max-agents cap + cost estimation before spawn                                                                              |
|**Data exfiltration via tools**       |Tool registry with risk levels; high-risk tools require governance approval per invocation                                                 |
|**Denial of service (self-inflicted)**|Rate limiting on heartbeat frequency; minimum interval between heartbeats per agent                                                        |

### 9.2 Output Sandboxing

Agent outputs are **never** trusted:

- Agent A’s output passed to Agent B is wrapped in `<agent_output source="agent_a">...</agent_output>` tags with explicit instructions to treat as untrusted data
- No raw output interpolation into system prompts
- All tool call arguments are validated against JSON Schema before execution
