## Overview

This module creates an Express Router and registers an async GET handler at /companies/:companyId/clipboard. The handler concurrently queries the database for company workspaces, recent workflow runs, active agent tasks, pending governance requests, non-terminal swarm runs, and recent heartbeat runs.

Results are fetched in parallel with Promise.all; workflow runs are further filtered in-memory to those belonging to the returned workspaces and with status "running" or "paused". Queries include ordering and limits (e.g., limit 50, heartbeat limit 10). Errors are forwarded to Express error middleware via next(err).

## Architecture Notes

- Parallel data fetching: uses Promise.all to execute six independent queries concurrently (workspaces, workflowRuns, agentTasks, governanceRequests, swarmRuns, heartbeatRuns) to minimize latency.
- MVP in-memory filtering: workflowRuns are queried globally (limited to 50) and then filtered in-memory to those whose workspaceId is in the returned workspaces set; comment notes company-scoping via workspace membership is applied in-memory (scalability trade-off).
- Terminal-phase exclusion: SWARM_TERMINAL_PHASES constant drives notInArray filter for swarm runs; the constant is typed as a readonly tuple and cast to any when passed to notInArray.

## Usage Examples

### Main use case

HTTP GET /companies/:companyId/clipboard — call with a companyId path parameter (string). The endpoint returns JSON with keys: workspaces (company's workspaces ordered by createdAt desc), workflows (filtered runs belonging to those workspaces with status 'running' or 'paused'), tasks (active agent tasks for the company), governance (pending governance requests), swarms (non-terminal swarm runs), and heartbeatRuns (recent heartbeat runs).

## Maintenance Notes

- Workflow scoping is applied in-memory after fetching workflowRuns. For large datasets or stricter correctness, move workspace join/filter into the DB query to avoid missing or excessive results.
- Hard limits and ordering are embedded in queries (limit 50 for most lists, 10 for heartbeatRuns). Review these limits for UX and performance implications and consider making them configurable.
- SWARM_TERMINAL_PHASES is cast to any when passed to notInArray — check the drizzle-orm typings and pass a correctly typed value to avoid runtime/type issues.
